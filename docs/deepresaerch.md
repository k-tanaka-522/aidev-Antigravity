Antigravity比較分析レポートエグゼクティブサマリー2025年後半、ソフトウェアエンジニアリングの現場は、従来の「コード補完（Autocomplete）」から、自律的な意思決定とタスク遂行能力を持つ「エージェンティック（Agentic）開発」へと急速なパラダイムシフトを迎えている。この変革の中心に位置するのが、Anthropic社が提供する Claude Code for VS Code と、Google社が発表した Antigravity である。本レポートは、これら二つのプラットフォームが提示する異なる技術的アプローチ、特に「サブエージェント（Sub-agents）」の構成、「オーケストレーション（Orchestration）」のメカニズム、そして「スキル（Skills）」およびツール統合のアーキテクチャについて、網羅的な調査と詳細な比較分析を行うものである。Claude Codeは、開発者が慣れ親しんだコマンドラインインターフェース（CLI）とVisual Studio Code（VS Code）の統合を深めることで、既存のワークフローを拡張する「進化型」のアプローチを採用している。対照的に、Google Antigravityは、IDEそのものをエージェントの管理と運用のために再構築した「革命型」のアプローチを採り、Gemini 3 Proモデルの推論能力を基盤としたマルチエージェント・オーケストレーション環境を提供する。本稿では、これらのツールがどのように開発者の役割を「コーダー」から「アーキテクト」へと変容させるか、その技術的特異性と実用性を詳細に論じる。第1章 序論：エージェンティック開発への移行とプラットフォームの哲学ソフトウェア開発ツールは、静的なテキストエディタから統合開発環境（IDE）、そしてAIによるチャット支援へと進化してきたが、現在の焦点は「自律性」にある。開発者が一字一句コードを記述するのではなく、高レベルの意図（Intent）を伝え、AIが計画・実行・検証を行うモデルである。この文脈において、Claude CodeとGoogle Antigravityは、異なる哲学に基づいた解を提供している。1.1 Claude Code: ターミナル中心の深化と拡張Claude Codeの本質的価値は、「ターミナルの中に住む熟練したペアプログラマー」 という概念にある 1。Anthropicは、開発者の作業の中心が依然としてファイルシステム操作やCLIコマンドの実行にあることに着目し、これをAIに代行させるアプローチを採用した。VS Code拡張機能としての提供は、ユーザーが既存のエコシステム（テーマ、キーバインディング、他の拡張機能）を維持できることを意味し、導入障壁を最小限に抑えている。その核となるのは、AnthropicのClaude 3.5 SonnetおよびOpusモデルであり、これらが持つ高度な推論能力と長大なコンテキストウィンドウ（200kトークン超）を活用して、大規模なコードベース全体を理解し、修正を加える能力である 1。1.2 Google Antigravity: エージェントファーストIDEの再定義一方、Google Antigravityは、「人間とエージェントが協調するためのミッションコントロール」 として設計されている 3。これはVS Codeのフォークでありながら、UI/UXは根本的に再設計されており、エディタ画面と同等かそれ以上に「エージェントマネージャー（Agent Manager）」と呼ばれる管理画面が重要視されている。GoogleのGemini 3 Proモデルをバックエンドに持ち、100万トークンを超えるコンテキストウィンドウと、ネイティブなマルチモーダル能力（テキスト、画像、動画の理解）を活かした開発体験を提供する。ここでの開発者の役割は、コードを書くことよりも、複数のエージェントにタスクを割り振り、生成された成果物（アーティファクト）をレビューすることに重点が置かれる 5。1.3 アーキテクチャの分岐点両者の決定的な違いは、「統合」対「再構築」 にある。Claude Codeは、OSのシェル（Bash/Zsh）と密結合し、ローカル環境のツールチェーンをそのまま活用する。対してAntigravityは、IDE内部に独自のランタイムとブラウザ環境を内包し、Google Cloud Platform（GCP）やAgent Development Kit（ADK）といったGoogleのエコシステムと深く連携することで、より管理された（Managed）環境を提供する傾向がある。以下の章では、このアーキテクチャの違いが、具体的な機能（サブエージェント、オーケストレーション、スキル）にどのように反映されているかを深掘りする。第2章 サブエージェント機能：専門化された労働力の構築と運用エージェンティック開発における「サブエージェント」とは、特定のタスクやドメインに特化したAIインスタンスを指し、メインのエージェントから切り離されたコンテキストや権限を持つことで、複雑な問題を効率的に解決する仕組みである。両プラットフォームはこの実装において、設定の柔軟性と実行モデルに大きな差異を見せている。2.1 Claude Codeにおけるサブエージェント：ペルソナとスコープの定義Claude Codeにおけるサブエージェントは、メインの会話スレッドから派生する「専門家」として機能する。これらは、特定の役割（Persona）と、使用可能なツールセット（Scope）によって定義される。2.1.1 JSON/YAMLによる構成管理Claude Codeのサブエージェントは、設定ファイル（~/.claude/agents/ またはプロジェクト固有の .claude/agents/）に配置されたMarkdownファイル（YAMLフロントマターを含む）によって定義される 7。このテキストベースの設定は、Gitによるバージョン管理と親和性が高く、チーム間での共有が容易である。具体的な構成要素は以下の通りである：名前と説明: エージェントを識別し、メインエージェントが「いつこのサブエージェントを呼び出すべきか」を判断するための自然言語による記述。システムプロンプト: エージェントの振る舞い、トーン、従うべきルール（例：「常にPEP 8準拠のコードを書くこと」「セキュリティ脆弱性を最優先でチェックすること」）を定義する。ツール制限: tools フィールドにより、そのエージェントがアクセスできる機能を制限できる。例えば、コードレビュー専用エージェントには Read、Grep、Glob の権限のみを与え、Edit や Bash の実行権限を剥奪することで、誤ってコードを破壊するリスクを排除できる 8。構成例（概念的再現）:YAML---

オーケストレーション：計画、実行、そしてフィードバックループ単発のコード生成ではなく、複数のステップにまたがる複雑なタスク（例：「全モジュールのテストを作成し、バグがあれば修正する」）を遂行するには、高度なオーケストレーション能力が必要となる。両プラットフォームは、このプロセスを制御するための独自メカニズムを備えている。3.1 Claude Code: シーケンシャル・プランニングと自律ループClaude Codeのオーケストレーションは、対話の流れの中で行われる 「計画（Plan Mode）」 と 「自律実行（Autonomous Loop）」 のサイクルに基づいている。3.1.1 Plan Mode（計画モード）複雑なリクエストを受け取った際、Claude Codeは即座にコードを書き始めるのではなく、「Plan Mode」に移行する 1。ここでは、タスクを達成するための詳細なステップ（例：1. 現状のコードを調査、2. 依存関係のインストール、3. ファイルAの修正、4. テスト実行）を列挙し、ユーザーに提示する。ユーザーはこの計画をレビューし、承認または修正を行う。この「合意形成」のプロセスが、手戻りを防ぐ重要なオーケストレーションの一部となっている。3.1.2 チェックポイントと巻き戻し実行フェーズにおいて、Claude CodeはGitの状態などを利用して自動的に チェックポイント を作成する 1。エージェントが試行錯誤の末にコードを破壊してしまったり、誤った方向に進んだ場合、ユーザーは /rewind コマンドを使用して以前の状態に戻すことができる。これは、AIの自律性を許容しつつ、安全網を提供する現実的な機能である。3.1.3 Hooksによるイベント駆動Claude Codeの隠れた強力な機能として Hooks がある 1。これは、特定のイベント（例：ツールの実行完了、エラー発生）をトリガーとして、あらかじめ定義したコマンドやスクリプトを実行させる機能である。使用例: 「コードの編集（Edit）が完了したら、自動的にリンター（Linter）を実行し、エラーがあればエージェントに再修正を指示する」といったループを構築できる。これにより、人間が介入せずとも品質を担保する自律的な修正サイクル（Self-healing）が実現する。3.2 Google Antigravity: 階層的タスク管理とInteractions
