# ドキュメント統括ワークフロー

## 概要
詳細設計フェーズで作成される複数の設計書を統括し、**DD000_詳細設計総括.md** を作成・更新します。

特に大規模システムで文書が分割された場合、全体を俯瞰する索引として重要な役割を果たします。

---

## 実行タイミング

以下のタイミングで実行：

1. **詳細設計開始時**: DD000の初期作成
2. **詳細設計書追加時**: DD000の更新（文書一覧に追加）
3. **詳細設計完了時**: DD000の最終確認・完成

---

## 入力確認

以下のディレクトリを読み込んでください：

```
templates/04_詳細設計/実成果物/
├── 01_アプリ/              # アプリチームの成果物
└── 02_インフラ/            # インフラチームの成果物
```

すべての詳細設計書（DD001-DD004, DD-INF001/002）をスキャンし、索引を作成します。

## DD000作成の手順

### 1. ディレクトリスキャン

```bash
# アプリチームの成果物
templates/04_詳細設計/実成果物/01_アプリ/

# インフラチームの成果物
templates/04_詳細設計/実成果物/02_インフラ/
```

### 2. 文書一覧の作成

各ファイルについて、以下を抽出：
- 文書ID（ファイル名から）
- 文書名
- 格納場所（相対パス）
- 担当（ファイル内のメタデータから、または「未定」）
- 状態（「完了」「作成中」「未着手」）

### 3. 文書間の関係マッピング

以下の関係を可視化：

```
RD003_機能要件定義書
  ├─ BD002_アプリケーション方式設計書
  │   ├─ DD001-01_クラス設計書_認証モジュール
  │   └─ DD003_API設計書_ユーザー管理API
  │
  └─ BD004_ネットワーク構成設計書
      └─ DD-INF001-01_Terraformスタック設計書_VPC
```

### 4. 進捗状況の可視化

```markdown
## 4. 進捗状況

### アプリケーション
- 完了: 5ドキュメント
- 作成中: 2ドキュメント
- 未着手: 1ドキュメント

進捗率: 62.5% (5/8)

### インフラストラクチャ
- 完了: 3ドキュメント
- 作成中: 1ドキュメント
- 未着手: 0ドキュメント

進捗率: 75% (3/4)
```

## 更新フロー

DD000は**living document**として、詳細設計フェーズ中に継続的に更新します：

1. **新しい詳細設計書が追加された場合**:
   - DD000の「2. 詳細設計書一覧」に追加
   - 進捗状況を更新

2. **詳細設計書の状態が変わった場合**:
   - DD000の「状態」列を更新
   - 進捗率を再計算

3. **詳細設計完了時**:
   - すべての文書が「完了」になっているか確認
   - 最終レビュー
